Claude Code · Deep Dive

Stop reading code.
Start seeing artifacts.

Artifacts are Claude Code's way of making output you can actually use — rendered UIs, persistent files, and interactive demos instead of wall-of-text responses. Here's how they work and when to reach for them.

5 min read · Developers new to Claude Code

01 · What

What is an artifact, exactly?

When you ask Claude a question, it responds in the chat. But some responses are more than answers — they're things: a component, a document, a working UI, a data file. An artifact is how Claude delivers those things as a proper, standalone output that renders, downloads, or runs.

Think of it as the difference between Claude describing a bar chart and Claude rendering one you can interact with.

Regular response

Chat output

Markdown text in the conversation. Useful for explanations, analysis, and short code snippets.

Artifact

Rendered output

A live HTML page, a downloadable file, a React component, or an SVG — usable immediately.

Artifacts can be HTML, React (.jsx), SVG, Markdown, or any file Claude writes to disk using its computer tools. In the Claude.ai interface they render inline. In Claude Code (CLI), they live as real files in your workspace.

Quick rule of thumb: if you'd copy the output out of chat and paste it somewhere else to use it, that's a sign it should be an artifact instead.

02 · How

How artifacts get created

In Claude Code, you trigger artifact creation through natural language. Claude decides the format based on your intent — but you can also be explicit.

1

You make a request with clear output intent

"Build me a dashboard," "write a report," "create a component" — the verb signals a file or rendered thing, not an explanation.

2

Claude picks the artifact type

HTML for UI/pages, React (.jsx) for interactive components, Markdown for documents, SVG for diagrams, plain files for everything else.

3

Claude Code writes the file to disk

In the CLI, the file lands in your project directory. Claude reads skills, checks constraints, then writes it — not just prints it.

4

You iterate in-place

"Make the chart blue," "add a loading state," "fix the padding" — Claude edits the artifact directly rather than rewriting from scratch.

In Claude Code's CLI you can also use /file commands and reference CLAUDE.md to give Claude instructions about how artifacts should be structured in your project.

CLAUDE.md (project root) markdown
# Project conventions for Claude Code ## Output paths - All generated UI goes in `src/components/generated/` - Reports go in `reports/` as Markdown ## Stack constraints - Vue 3 + TypeScript (no React) - Tailwind for styling - No external chart libraries — use native SVG

Claude reads CLAUDE.md at session start and applies these rules to every artifact it creates. This is the most reliable way to keep generated files consistent with your actual codebase.

03 · Use cases

What to build with artifacts

Artifacts earn their place when the output needs to be used somewhere — not just read. Here are the most common patterns:

⚡

Instant UI prototypes

Spin up a working page or component in seconds. Great for stakeholder demos and design reviews before you wire up real data.

"Build a Tailwind Vue card component for a CRM lead record"
📊

Data visualisations

Give Claude a CSV or schema and get back a live chart — recharts, D3, or plain SVG — ready to embed.

"Chart this CSV data as a stacked bar grouped by category"
📄

Structured documents

Generate Confluence-ready Markdown, Word docs, or PDFs with real content — not templates to fill in later.

"Write a Confluence page documenting this n8n workflow"
🗺️

Architecture diagrams

Describe a system in prose and get back a Mermaid or SVG diagram. Update it by describing what changed, not redrawing from scratch.

"Diagram the n8n → AI model → database pipeline as a flow chart"

04 · Best practices

Tips for getting good artifacts

✅

Be output-explicit

Say "create a file," "write a component," or "build a page" — not just "help me with." Claude needs a signal that you want a standalone output, not chat prose.

✅

Put constraints in CLAUDE.md, not every prompt

Framework, stack, output paths, naming conventions — these belong in your project's CLAUDE.md so you don't repeat them. Claude checks it at session start.

✅

Iterate on the same artifact, don't regenerate

"Update the chart to use teal" is better than re-prompting from scratch. Claude uses str_replace to edit files in-place, keeping your iteration loop tight.

✅

Ask for the artifact to run itself

In Claude Code, you can ask Claude to write a script and execute it, then fix any errors. Close the loop: "write the migration, run it, report what changed."

⚠️

Don't use artifacts for short, throwaway snippets

A 5-line helper function or a one-off SQL query doesn't need to be a file. Artifacts add overhead — save them for things you'll actually keep or share.

React artifacts only: localStorage and sessionStorage don't work inside Claude.ai's sandboxed artifact renderer. Use React state (useState) for in-session data. If you need persistence across sessions, you'll need to take the code into your own environment.
