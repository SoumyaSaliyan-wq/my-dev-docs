Claude Code · MCPs

Build your own MCP.
Wire Claude into anything.

Existing MCP servers cover GitHub, Notion, Atlassian. But what about your internal APIs, private databases, or custom tooling? Here's how to build one from scratch — schema, handlers, and all.

8 min read · Intermediate — TypeScript assumed

01 · Why

When off-the-shelf MCPs aren't enough

MCP (Model Context Protocol) is an open standard — any AI tool can implement it, and a server you build today could in theory work with Claude, ChatGPT, or whatever comes next. This post focuses specifically on using it with Claude Code, but the server-side concepts apply broadly.

The MCP ecosystem covers a lot of common ground — GitHub, Slack, Atlassian, Notion, Figma. But most teams run internal tools that nobody has written an MCP for. Your staging API. Your company's data lake. A custom search index. A deployment script that only lives on your infra.

Building your own MCP server is how you give Claude actual access to those things — not just the ability to talk about them in theory, but to call them during a conversation the same way it calls any other tool.

New to MCPs? This post assumes you already know what MCPs are and why they're useful. If not, start with Give Claude Code actual access to your stack first, then come back here.

02 · Anatomy

What an MCP server is made of

An MCP server is a small process that speaks the Model Context Protocol. Claude Code (the client) connects to it and can call anything it exposes. Servers can expose three things:

🔧

Tools

Functions Claude can call with arguments. This is the 90% case — "search tickets," "deploy branch," "get user record." Tools return text or structured data.

📄

Resources

Read-only content Claude can fetch by URI — docs, configs, file contents. Think of them as context Claude can pull in on demand rather than you pasting it manually.

💬

Prompts

Reusable prompt templates with parameters. Less common — mainly useful for slash-command style interactions inside Claude Desktop.

For most internal tools, you only need tools. Resources and prompts are optional. Start there.

On transport: MCP servers talk to clients over stdio (standard input/output, used by Claude Code CLI) or SSE (Server-Sent Events, used by remote servers). Building locally, use stdio — it's simpler and requires no network setup.

Start here

stdio

Claude Code spawns your server as a child process. Communication happens over stdin/stdout. Zero networking, no auth needed during dev.

For remote servers

HTTP + SSE

Your server runs as an HTTP service. Claude connects to it over the network. Required when you want to share an MCP across a team or deploy it centrally.

03 · Build

Building a server step by step

The pattern is the same regardless of what you're wrapping — a REST API, a database query, a CLI tool, or a proprietary SDK. Here's the sequence.

1

Scaffold the project

Create a new Node.js project and install the official MCP SDK (@modelcontextprotocol/sdk). The SDK handles all the protocol-level communication so you only write application logic. TypeScript is the recommended choice — the SDK ships with types and the tooling is well-supported.

2

Declare your tools

Every MCP server needs a list handler — a function that tells Claude what tools exist and what each one expects. For each tool you define a name, a plain-English description, and a JSON Schema for its arguments. This is the contract between your server and Claude.

The description is the most important field. Claude reads it to decide whether to call the tool at all. Write it like you're explaining the tool to a smart colleague who can't see your code — what does it do, what does it return, when should they reach for it.

3

Handle tool calls

The second handler you always write is the call handler. When Claude decides to invoke one of your tools, this runs. It receives the tool name and the arguments Claude picked, does the actual work — fetches an API, queries a database, runs a command — and returns the result as text.

Keep each branch of your handler focused. If you notice a single handler getting long, that's usually a sign your tools aren't narrow enough. Split them.

4

Connect a transport

The last step is attaching a transport so Claude can actually reach your server. For local development, use stdio — Claude Code spawns your server as a child process and communicates over stdin/stdout. No networking required. When you're ready to share the server across a team, swap in an HTTP+SSE transport and deploy it as a service.

5

Register it in Claude Code

Claude Code has built-in support for adding and managing MCP servers — the exact config location varies by platform and install method, so follow the official Claude Code MCP docs for the current steps. In general you'll point Claude at the command to start your server and pass any secrets as environment variables — never in source code.

Once registered, run /mcp in the chat to confirm the server shows up as connected.

04 · Auth

Locking down access — SSO, tokens, and roles

Auth requirements depend entirely on how your server runs. The two transports have very different threat models.

stdio — local only

OS-level isolation

Claude Code spawns the server as a child process under your OS user. No network exposure. Auth comes for free from the OS — only your account can start the process.

HTTP + SSE — remote

Explicit auth required

Your server is reachable over the network. Anyone who can reach the endpoint can call your tools. You need to check identity on every request.

For teams sharing a remote MCP server, the two most common patterns are API key auth (simple, token passed in a header) and OAuth 2.0 / SSO (recommended for anything touching sensitive data or needing per-user identity).

The role check happens inside your call handler — after you've validated the token with the identity provider, you read the user's roles or scopes from the token claims and decide whether that particular tool call is allowed. Reject early and return an error; don't partially execute and then fail.

For API key auth: pass the key in the Authorization: Bearer <token> header on every request. Validate it against a lookup (database, secrets manager, or env-based allowlist) before running any tool logic. Simpler than OAuth but no per-user identity — every caller looks the same to your server.

05 · Debug

Testing before you connect Claude

You don't need Claude running to test your MCP server. The MCP Inspector (@modelcontextprotocol/inspector) gives you a browser UI to call your tools directly and see exactly what they return — browse your tool list, fill in arguments, inspect the raw response. Run it with a single npx command pointed at your server entry point.

Fix your schema and handlers here before wiring Claude in. It's much faster to iterate in the inspector than through a conversation.

stdio means no console.log debugging. Your server's stdout is the protocol channel — anything you write there that isn't valid MCP JSON will corrupt the connection. Use process.stderr.write() or a file logger during development instead.

When you do connect Claude, use /mcp to check server status and watch the terminal where you ran Claude Code — MCP errors surface there.

06 · Tips

Things that make a real difference

01

One tool = one operation

Don't build a mega-tool with a action: "get" | "list" | "create" parameter. Claude picks tools by reading their descriptions — if one tool does three things, Claude has to guess. Three focused tools with clear descriptions win every time.

02

Return structured JSON, not prose

Your tool returns a content array. Put JSON.stringify(data, null, 2) in the text field. Claude can reason over structured data much better than a formatted string you wrote. Let Claude decide how to present it to the user.

03

Always handle errors explicitly

If your API call fails or returns a non-2xx, return an error object in the content — don't let it throw uncaught. An unhandled exception crashes the tool call silently from Claude's perspective. Return { isError: true, content: [{ type: "text", text: err.message }] } so Claude can reason about the failure and tell the user what went wrong.

04

Put secrets in env, not code

Use the env block in the MCP config for API keys and tokens. Read them with process.env in your handler. This keeps credentials out of your source tree and makes it easy to use different tokens per environment.
